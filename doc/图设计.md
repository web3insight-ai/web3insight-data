                 +-----------------+

## 图数据库 构建方案



### 节点定义

| **节点类型**           | **唯一属性** | **其他属性**                                                 | **含义**           |
| ---------------------- | ------------ | ------------------------------------------------------------ | ------------------ |
| **Developer** (Actor)  | actor_id     | actor_login, created_at                                      | GitHub 开发者      |
| **Repository** (Repo)  | repo_id      | repo_name, eco_names, is_abnormal, created_at, custom_marks, upstream_marks | 代码仓库           |
| **Organization** (Org) | org_id       | org_login                                                    | 组织（GitHub Org） |

### 边定义

|          **边类型**          | **起点**   **→** **终点**     | **属性**                                        | **含义**             |
| :-----------------------: | ------------------------- | --------------------------------------------- | ------------------ |
|    **TRIGGERED_EVENT**    | Developer → Repository    | event_type, payload, body, created_at, public | 开发者在某仓库触发了某种事件     |
| **MARKED_AS_UPSTREAM_OF** | Repository → Repository   | 来源于 upstream_marks                            | 一个仓库被标记为另一个仓库的上游项目 |
|      **BELONGS_TO**       | Repository → Organization | 无                                             | 仓库属于某个组织           |

### 分析

- 现在通过**边**就可以分析获得开发者的**行为数据**，但是内容信息仍需要获取边中的具体属性进行分析。

- 这种方案下，是否属于一个生态，是依赖仓库仓库间上下游，但是如何确定仓库中心点？（我这里倾向于指定生态中心仓库的思路）
- 针对于某个repo 是否在某个生态中，是否边缘问题，如何衡量边缘距离？（如果在中心仓库确定的情况下，可以使用`MARKED_AS_UPSTREAM_OF`这个边的多跳次数，但是不确定在现有的数据中，是否会出现比较混乱的上下游关系，相互fork的问题）
- 第三类节点和第三类边是否有必要，存疑，可以没有，也可以有，感觉对于咱们的分析可能贡献不大，但是删除的话，可以在这个图数据库中，会丢失部分信息
- 对于不同类型的边，可以在混合查询时，赋予不同的权重，或者根据路径衰减，进而体现权重（比如更侧重repo-repo的贡献，或者更侧重开发者-repo的贡献等）
- 现在有多种节点和多种边，非同质信息网络，信息量大，分析复杂


## openrank算法

五种主要事件为 Issue 评论、新建 Issue、新建 PR、PR Review 评论和 PR 合入

通过加权求和的方式 获取开发者活跃度

项目活跃度是，所有开发者活跃度的开方总和

项目间关联度是通过同时活跃过的开发者进行计算的，计算所有同时活跃开发者的调和平均数
